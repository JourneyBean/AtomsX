## Why

当前前端没有统一的错误反馈机制。用户操作（如创建工作空间、发送消息）失败时，错误信息仅打印到 console.error，用户无法感知发生了什么问题。这导致用户体验不佳——点击按钮后若无响应，用户不知道是网络问题、权限问题还是其他原因。

在 MVP 阶段，用户需要流畅的"边聊边改边预览"体验，错误反馈是必不可少的基础能力。

## What Changes

- 新增全局 Toast/Notification 组件，显示操作结果反馈
- 新增 Notification Store，统一管理通知状态
- 在 API 调用失败时自动弹出错误提示（右上角）
- 支持成功/警告/错误三种通知类型
- 通知自动消失（可配置持续时间）
- 用户可手动关闭通知

## Capabilities

### New Capabilities
- `frontend-notification`: 全局通知/Toast 系统，用于显示操作结果、错误提示等信息反馈

## Impact

- **前端代码**：新增 NotificationStore、Toast 组件，修改现有 API 调用逻辑以集成错误反馈
- **依赖**：无新增外部依赖，使用纯 Vue 3 + Pinia 实现
- **影响层级**：纯控制面前端，不影响 Workspace Runtime、预览网关、构建器、部署器

## Non-goals

- 不做通知持久化（刷新页面后通知消失）
- 不做通知历史记录页面
- 不做桌面推送通知
- 不做后端错误码映射表（第一阶段直接显示后端返回的错误信息）

## Security & Multi-tenancy

- 无安全边界影响
- 无多租户隔离影响
- 通知内容仅显示给当前用户，不涉及跨用户共享

## User Value

- 用户能及时了解操作失败原因，减少困惑
- 提升产品可用性和用户信任度
- 为后续更完善的错误处理（如错误码标准化）奠定基础
